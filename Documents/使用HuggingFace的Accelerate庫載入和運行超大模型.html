載入和運行超大模型

隨著超大規模語言模型的發佈（沒錯就是GPT3，以及最近很火的Chatgpt），Meta AI和BigScience也發佈了大量開放原始碼的大規模語言模型，這些模型完全超出了大多數人的硬體載入能力（記憶體不夠，GPU視訊記憶體也不夠）。在Huggingface，提供了一些好用的工具，無需超級電腦，你就可以載入和運行大規模語言模型。本部落格翻譯上圖作者，下面的程式碼可以在Colab或者kaggle的kernel上運行，在學習本教學後，大家可以嘗試在自己的電腦上載入和運行大模型。

下面我們展示如何運行OPT-6.7B模型：

import torch
from transformers import pipeline
​
# 我們在Kaggle的kernel上運行
# 選擇一個大模型，當然需要足夠多的硬碟空間，並且要耐心等待
checkpoint = "facebook/opt-6.7b"
generator = pipeline("text-generation", model=checkpoint, device_map="auto", torch_dtype=torch.float16)
​
# 使用pipeline進行推理
generator("More and more large language models are opensourced so Hugging Face has")

使用pytorch的傳統pipeline方式來載入模型，一般包括以下步驟：

  1. 建立模型
  2. 將權重載入到記憶體（一個叫做state_dict的字典對象）
  3. 在已建立模型基礎上載入權重值
  4. 將模型載入到相應裝置上（比如GPU）進行模型推理

如果模型不是很大的話，使用上面的程式碼載入權重是可以的。上面的模型有67億個參數，使用默認精度儲存這些參數的話，僅上面的第一個步驟，就要使用26.8GB的記憶體容量（在記憶體中默認float32格式的1個參數佔據4個字。節），如果使用Colab或者kaggle的kernel很可能會報錯。

第二步將模型載入到記憶體中（在使用默認精度格式下，消耗26.8GB記憶體）。如果載入更大的模型，比如BLOOM或OPT-176B（有1760個參數），需要大概1.4TB的CPU記憶體空間，很好的電腦也沒有這麼大的CPU記憶體呀！如果還要匯入GPU那麼就更麻煩了！

因此我們需要做一些技巧性的改變（時間換空間！）本文，將介紹，在有限記憶體和視訊記憶體資源的情況下，如何使用accelerate庫來完成這些超大模型的載入和推理。操作步驟如下：

  1. 建立空模型（沒有權重）
  2. 設定哪一層進行推理（以及設定運行所在的裝置）
  3. 載入部分權重到記憶體
  4. 將記憶體中的部分權重載入到空模型中
  5. 將權重傳到裝置上完成推理
  6. 載入剩下部分的權重，重複步驟3，直到載入所有剩餘權重，並完成最終推理

建立空模型

PyTorch 1.9版本提供了一個叫做meta的新裝置。使用該裝置，可以方便建立模型，而無需載入權重。使用該裝置，可以建立非常大的張量，而無需考慮記憶體或視訊記憶體的大小。

例如，下面的程式碼如果在Colab上或kaggle的kernel運行，會報錯（因為OOM，out of memory，超出儲存能力）。

import torch
​
large_tensor = torch.randn(100000, 100000)

上述所使用torch建立的張量，需要4*10**10個位元組（默認使用精度為32位的情況下，張量為1的話，需要佔據4個位元組儲存空間），這個定義的張量需要40GB的記憶體（對於一般的電腦或機器，會報錯，讀者的記憶體有多大呀！）。如果使用如下程式碼，只在參數定義中設定device=meta即可只定義張量形狀，而不消耗記憶體。

import torch
​
large_tensor = torch.randn(100000, 100000, device="meta")

可以嘗試print上面的張量，結果如下。

tensor(..., device='meta', size=(100000, 100000))

正如前述，上面的張量只有形狀，並沒有資料（不佔據記憶體或視訊記憶體）。

當然，也可以使用meta來建立模型。

large_model = torch.nn.Linear(100000, 100000, device="meta")

但是對於已有模型，使用上述方法來重寫模型，顯然是不明智的，尤其是對Transformers庫中這成千上萬的模型。Huggingface提供了一個上下文管理器，來使用meta初始化一個空模型（只有shape，沒有資料）。

下面程式碼用來初始化一個BLOOM空模型。

from accelerate import init_empty_weights
from transformers import AutoConfig, AutoModelForCausalLM
​
config = AutoConfig.from_pretrained("bigscience/bloom")
with init_empty_weights():
    model = AutoModelForCausalLM.from_config(config)

上面的程式碼也可以用來初始化其它的空模型，但是這裡返回結果是不能進行模型計算。另外要注意的不是模型的所有操作都被初始化為meta。定義好上述模型後，可以喂輸入，然後得到一個meta裝置的輸出張量（同樣，只有形狀，沒有資料，其實就是進行了shape計算）。

huggingface還在研發一個稱為FakeTensor的類，與meta裝置操作很像，但是會包含裝置資訊（同時有形狀和資料類型，比meta多了個dtype輸出）

從上面操作，可以知道每個權重的形狀大小，然後就可估算出載入預訓練權重需要的記憶體大小。因此下步為了適配有限記憶體和視訊記憶體，我們需要合理的切分模型，進行CPU和GPU計算操作。

計算裝置圖

在開始載入預訓練權重前，我們需要知道將它們放到模型中的哪個部分。在每次載入新權重的時候的可以釋放掉前一階段載入權重，以釋放記憶體。這個操作可以通過meta裝置來實現，在這之前我們已經知道了每個張量的形狀帶下和資料類型，並且通過這些資訊可以方便的估算出消耗記憶體的大小。

Accelerate庫提供了一個函數用來自動檢測一個空模型使用的裝置類型。它會最大化利用所有的GPU資源，然後再使用CPU資源（還是遵循速度快的原則），並且給不匹配的權重（硬碟上的離線載入權重）打上標記。下面看下載入OPT-13b模型（有130億個參數）。

from accelerate import infer_auto_device_map, init_empty_weights
from transformers import AutoConfig, AutoModelForCausalLM
​
config = AutoConfig.from_pretrained("facebook/opt-13b")
with init_empty_weights():
    model = AutoModelForCausalLM.from_config(config)
​
device_map = infer_auto_device_map(model)

Accelerate評估embeddings和decoder，在第9個decode之前，都市在GPU上運行，但是從第10個decode開始到第17層，都是權重部分在CPU，另一部分在GPU上。然後第18層權重，部分在CPU上，部分在硬碟上（硬碟上的資料就是離線載入的，所謂的offload）。

實際上上面的裝置對應其實沒有什麼用，因為在實際的模型計算時還有很多殘差連接（經過處理的結果和輸入進行相加操作），因此所有的層都要在相同的裝置上。可以通過設定no_split_module_classes來設定（傳入一個模型名稱列表），如下所示（下面的OPTDecoderLayer從哪裡來，為什麼就可以起到效果）。

device_map = infer_auto_device_map(model, no_split_module_classes=["OPTDecoderLayer"])

現在層不會被切分到不同的裝置上了。

在Transformers庫中，在from_pretrained()或這pipeline()函數中設定device_map，則會自動完成上面的操作，因此設定參數no_split_module_classes就沒那麼麻煩了（還真是HuggingFace一貫的風格，矮油，不錯！）。device_map可以有如下選項可供設定（如果有多個GPU又是另外一回事了，那就是模型平行了）。

  • "auto" 或 "balanced": Accelerate將會根據所有GPU均衡切分權重，儘量均勻的切分到各個GPU上；
  • "balanced_low_0": Accelerate均勻分割權重到各個GPU上，除了第一個GPU（序號為0）。在第一個GPU上會儘量節省視訊記憶體（這個模式方便在第一個GPU上留夠資源進行注入模型輸出計算等，例如使用generate函數，這種模式可以有效節省第一個GPU的視訊記憶體用於模型生成操作）；
  • "sequential": Accelerate按照GPU的順序佔用視訊記憶體（因此排序好後的GPU視訊記憶體佔用會少一些）。

可以從上面的選項中按照自己的需求設定device_map參數（也可以使用layer/module的格式來設定no_split_module_classes參數）。

最後device_map的元素也可以設定為不同的資料格式（不同資料格式，導致記憶體和視訊記憶體空間佔用也不同，例如常用float32佔用4個位元組，而int8則只佔用1個位元組）。如下所示，設定dtype="float16"，會得到不同的結果。

device_map = infer_auto_device_map(model, no_split_module_classes=["OPTDecoderLayer"], dtype="float16")

現在知道了每一層是在哪個裝置上運行，後面就可以向指定層載入權重了。

state dicts的分片

傳統的，Pytorch會保存一個完整的檔案，包含參數名到權重的對應關係。這個對應關係表稱為state_dict。下面是PyTorch文件 給出的保存和載入模型程式碼示例。

# Save the model weights
torch.save(my_model.state_dict(), 'model_weights.pth')
​
# Reload them
new_model = ModelClass()
new_model.load_state_dict(torch.load('model_weights.pth'))

在模型參數小於10億時，上面程式碼一般運行沒有問題。但是對於大模型（超過10億參數），上面的程式碼就有點不能用了。像BLOOM模型有1760億個參數，就算使用bfloat16資料類型來保存權重，它仍然會消耗352GB的儲存空間。對於超級電腦，可以完成載入、訓練和保存的任務，但是對於使用者側的推理，這個儲存需求就要另外想辦法了。

上面講述了也是為什麼Hugging Face Hub的大模型權重都是被劃分成多個檔案來儲存。例如，打開連結BLOOM模型頁 ，你會發現72個檔案名稱（格式為pytorch_model_xxxxx-of-00072.bin），都是模型權重的一部分。通過這種檔案格式切分，可以在記憶體中一個一個載入（不用全部載入，導致超出記憶體儲存能力。可以實現載入部分權重到模型、移動到GPU或其他裝置上進行計算、以及載入下一部分權重時，去除上一部分已載入的權重來釋放記憶體或視訊記憶體等）。與載入全部的模型不同，我們的機器只需要能夠載入最大的一個checkpoint即可，這裡稱為分片，例如BLOOM，留夠儲存空間載入7.19GB即可（也挺大的！！）。

如果一個模型權重檔案被切分為多個檔案，例如BLOOM分片checkpoint，我們可以格式標準化如下：

  • 一個檔案（稱為pytorch_model.bin.index.json）包含一些中繼資料以及一個參數名對應表，包含權重指示資訊。
  • 所有其他的檔案是標準的Pytorch state dicts，它們各自包含模型的一部分。可以在這裡找到一些索引資訊。

載入上面例如BLOOM這種模型，我們需要循環載入各個分片。Accelerate提供了一個函數load_checkpoint_in_model可以完成該操作。使用者可以載入本地的checkpoint或者使用from_pretrained來載入Transformers模型。

import torch
from transformers import AutoModelForCausalLM
​
# 會報錯
checkpoint = "facebook/opt-13b"
model = AutoModelForCausalLM.from_pretrained(checkpoint, device_map="auto", torch_dtype=torch.float16)

如上所示，如果device map被設定為自動模式，且讀者機器沒有足夠的記憶體會視訊記憶體，那麼上面的程式碼會報錯，並且提示你需要設定一個資料夾來保存那些離線載入到硬碟的權重。

ValueError: The current `device_map` had weights offloaded to the disk. Please provide an 
`offload_folder` for them.

解決上面的問題，可以設定參數offload_folder="offload"和torch_dtype=torch.float16。

import torch
from transformers import AutoModelForCausalLM
​
# 如果在Colab上運行，會報錯（OOM，記憶體不夠）
checkpoint = "facebook/opt-13b"
model = AutoModelForCausalLM.from_pretrained(
    checkpoint, device_map="auto", offload_folder="offload", torch_dtype=torch.float16
)

注意到如果嘗試載入一個非常大的模型，該模型需要很多的離線的硬碟空間，那麼上面的程式碼在載入完最後一個checkpoint分片之後會包超出記憶體錯誤。可以通過設定參數offload_state_dict=True來臨時離線硬碟載入部分模型，並且在其他權重處理完之後，多載這部分權重到記憶體。

import torch
from transformers import AutoModelForCausalLM
​
checkpoint = "facebook/opt-13b"
model = AutoModelForCausalLM.from_pretrained(
    checkpoint, device_map="auto", offload_folder="offload", offload_state_dict = True, torch_dtype=torch.float16
)

上面的程式碼在Colab上運行良好，但是也就只夠載入模型，如果想做模型推理或其它操作，那麼記憶體就不夠用了。為了得到一個可以正在用來推理的模型，我們需要在影片上離線載入更多的層。可以通過傳入device_map參數（前面已經介紹）來解決該問題，如下所示。

import torch
from transformers import AutoModelForCausalLM
​
checkpoint = "facebook/opt-13b"
device_map["model.decoder.layers.37"] = "disk"
model = AutoModelForCausalLM.from_pretrained(
    checkpoint, device_map=device_map, offload_folder="offload", offload_state_dict = True, torch_dtype=torch.float16
)

模型推理：在若干裝置上運行模型分片

最後一個步驟就是如何使用Accelerate來實現模型逐步載入權重到若干GPU、CPU和硬碟，並推理。這個可以通過hook鉤子操作來實現

hooks是一個Pytorch的API，該鉤子函數會在每個前饋操作前呼叫

這個操作一般使用如下程式碼完成（具體還沒有測試），然後就可以把model當做一般模型來使用了，比如要推理的話，就可以使用model.generate()來生成文字：

from accelerate import load_checkpoint_and_dispatch
​
model = load_checkpoint_and_dispatch(
    model, "facebook/opt-13b", device_map=device_map, no_split_module_classes=["OPTDecoderLayer"]
)

有三個需要注意的地方：

  • 確保子模組的資料和權重在相同的裝置（device）上
  • 如果權重離線載入到CPU（記憶體），將他們移動到GPU 0上，然後再進行前饋計算，然後將結果返回到CPU上
  • 如果權重被離線載入到硬碟，將他們移動到記憶體，然後移動到GPU 0，再進行前饋計算，最後釋放記憶體。

完成的模型分片推理過程在連結https://www.youtube.com/embed/MWCSGj9jEAo上，國內無法訪問。

通過上面的一系列操作，我們就可以在沒有充足記憶體和視訊記憶體的條件下，完成模型的載入和推理操作。當然需要有足夠的硬碟空間（典型時間換空間，空間指的是記憶體和視訊記憶體空間，用硬碟空間代替了，有點類似記憶體對應技術）。當然現在的方法還是很初級的（直接使用多個GPU，當然更聰明的方法還有使用GPU序列操作，可以見連結pretty decent results for BLOOM）。另外使用Accelerate進行大模型的推理，可以看連結 accelerate文件。
