文字翻譯

下面我們看下如何使用transformers庫來進行翻譯模型的訓練及相關操作。可以認為翻譯是另外一種sequence-to-sequence seq2seq任務，該任務將一個序列通過某種變換成另外一個序列。這個問題和摘要任務非常相似（摘要是將一個較長的文字轉換為較短的文字）。本節也會介紹一些其他的seq2seq問題。

  • 風格轉換Style transfer: 建立一個模型，將一种放歌的文字轉換為另外一種風格的文字（例如將傳統或莎士比亞風格的英語轉換為現代英語）
  • 生成式問答: 建立一個模型，根據一段文字，來生成一個問題的答案。

如果讀者有海量兩個成對或多個成對的文字語料時，你就可以訓練類似causal language modeling這種自編碼模式的新型翻譯模型（ChatGPT就是這麼做的）。這樣的模型速度會比較快，但是如果我們要微調一個現有的翻譯模型，例如將多語言翻譯模型如mT5或mBART微調為只對一個指定語言對進行翻譯，或者將一個指定翻譯模型，將一種語言翻譯為另外一種你語料庫對應的語言。（這一段好神奇，但是GPT的神奇能力相比大家在ChatGPT上已經看到了）

在本節中，我們微調一個預訓練的Marian模型，實現從英文到法文的翻譯（Hugging Face的很多僱員大多使用這兩種語言）。資料集為KDE4 dataset，該資料集來自於KDE apps。該模型已經在大型法語和英文語料庫上進行了預訓練，該語料庫為Opus dataset，該資料集包含KDE4資料集。儘管我們的模型已經海量資料集上進行了預訓練，但是我們依然可以通過微調獲得更好的效果。

一旦完成了訓練，我們可以通過gradio來建構我們的應用，具體參考huggingface上例子。

​ 與前面章節類似，我們訓練的模型和上傳的模型，可以查看連結here.

準備資料

為了微調或從頭訓練一個翻譯模型，我們需要一個合適的資料集（啥才能稱為合適？）。如前所述，我們在本節中使用KDE4資料集 ，另外通過微調下面的程式碼可以很方便使用讀者自訂的資料集，只要該資料集是按照成對的翻譯文字出現即可（當然要滿足一定的結構，具體編碼時就知道了）。如果想瞭解更多關於自訂資料集載入的知識，可以複習第5章 內容。

準備資料

輸入到模型的資料都必須是整數，也就是需要將文字轉換為token IDs的集合，模型才能夠進行計算，並得到結果。對於翻譯任務，需要對輸入和標籤都進行分詞處理。我們首先建立一個tokenizer對象。這裡我們使用一個已經在英文轉法文資料集上做過預訓練的模型。如果需要處理其它的語言，則要相應的調整你的模型 checkpoint。Helsinki-NLP這個開源組織提供了很多的語言對預訓練模型（可以打開連結看看）。

 from transformers import AutoTokenizer
 ​
 model_checkpoint = "Helsinki-NLP/opus-mt-en-fr"
 tokenizer = AutoTokenizer.from_pretrained(model_checkpoint, return_tensors="tf")

可以從 Hub替換你需要的model_checkpoint，或者從本地載入你自己預訓練的模型和分詞器。

如果使用多語言分詞器，例如mBART、mBART-50或M2M100，你需要通過設定tokenizer.src_lang和tokenizer.tgt_lang來指定輸入和輸出的分詞器。

進行分詞的時候，需要注意的是，在對輸入進行分詞時，就和前面章節的操作類似。但是對於目標文字進行分詞時，需要加上上下文管理器as_target_tokenizer()。（當然，如果輸入和輸出是同一種語言的話，就不需要了，因為輸入和標籤用的是同一種分詞器）

Python中的上下文管理器，使用with關鍵詞（在操作磁碟檔案的時候經常用到），如下所示：

 with open(file_path) as f:
     content = f.read()

上面已經隱含了檔案的打開和關閉操作（程式碼可讀性提高了）。打開檔案的控制代碼f在with包含的語句片段有效，離開後會關閉控制代碼並釋放資源。

在本例中，我們在對輸入進行分詞操作時，呼叫with改變分詞器為法文分詞器，在法文文字分詞結束後回到英文分詞器，程式碼如下所示（下面為對一個文字對進行操作的例子）。

 en_sentence = split_datasets["train"][1]["translation"]["en"]
 fr_sentence = split_datasets["train"][1]["translation"]["fr"]
 ​
 inputs = tokenizer(en_sentence)
 with tokenizer.as_target_tokenizer():
     targets = tokenizer(fr_sentence)

如果沒有使用with進行分詞器轉換，那麼對目標也會使用英文分詞器，結果如下（不是法文分詞，同是拉丁語系，可能還好，但是如果不同語系，也許會得到很差的結果，也許吧）。

 wrong_targets = tokenizer(fr_sentence)
 print(tokenizer.convert_ids_to_tokens(wrong_targets["input_ids"]))

 ['▁Par', '▁défaut', ',', '▁développer', '▁les', '▁fils', '▁de', '▁discussion', '</s>']

如上所屬，英文分詞會把法語句子分詞為非常多的token，而且有些法文單詞沒有被分詞器認出來。

inputs和targets是python字典，包含input_ids、attention_mask和token_type_ids等鍵，需要設定labels鍵給模型輸入。這個事情在預處理函數中完成，如下所示。

 max_input_length = 128
 max_target_length = 128
 ​
 ​
 def preprocess_function(examples):
     inputs = [ex["en"] for ex in examples["translation"]]
     targets = [ex["fr"] for ex in examples["translation"]]
     model_inputs = tokenizer(inputs, max_length=max_input_length, truncation=True)
 ​
     # Set up the tokenizer for targets
     with tokenizer.as_target_tokenizer():
         labels = tokenizer(targets, max_length=max_target_length, truncation=True)
 ​
     model_inputs["labels"] = labels["input_ids"]
     return model_inputs

注意這裡將輸入和輸出設定了相同的最大token長度，因為我們資料集中的文字都比較短，這裡使用128。

提示：如果使用T5模型，模型希望輸入文字提供一個前綴，對於翻譯模型，新增translate: English to French:。
提示：這裡不關注標籤的掩碼attention_mask，因為模型不需要（但是計算標籤精準率的時候，可以用這個mask）。不用attention_mask的一點是，我們在設定標籤的時候，會將padding部分使用-100代替，-100在計算損失函數的時候會忽略掉這些padding部分。這部分操作會在後面的data collator函數中完成，我們在data collator中進行動態padding，即只針對一個batch進行padding。如果在上面的預處理函數中進行padding，則必須對labels中對應的padding部分進行賦值-100操作（這是靜態padding了）。

下面使用map函數完成對資料集中所有資料的預處理。

 tokenized_datasets = split_datasets.map(
     preprocess_function,
     batched=True,
     remove_columns=split_datasets["train"].column_names,
 )

現在這些資料已經被預處理了，現在開始微調預訓練模型。

使用TrainerAPI進行模型微調

和之前章節類似，使用Trainer的程式碼類似，但是有一點點小區別，就是我們這裡使用Seq2SeqTrainer 。該類是Trainer的繼承類，允許我們在合適的處理驗證操作，即使用generate()函數來根據輸入預測輸出。當討論指標計算的時候，會深入聊下這個新類。

首先，我們需要載入和快取一個實際模型，使用AutoModel API，如下：

 from transformers import AutoModelForSeq2SeqLM
 ​
 model = AutoModelForSeq2SeqLM.from_pretrained(model_checkpoint)

這裡我們使用一個在翻譯任務已經預訓練的模型，因此載入後不會警告說有新初始化的權重。

Data collation

在動態batching中我們需要一個data collator完成padding。這裡不適用DataCollatorWithPadding來進行補齊操作，因為這個函數僅對輸入的鍵（包括input_ids, attention_mask, token_type_ids）進行補齊，不會對labels進行補齊操作。還有在對labels進行補齊操作時，使用的是-100而不是分詞器的pad_token，這麼做到的目的是在計算損失函數的時候忽略掉這些padding token。

DataCollatorForSeq2Seq完成上述操作（還真是方便，但是建議讀者還是進入原始碼，看下具體的操作，這樣使用的時候會更有信心）。預DataCollatorWithPadding類似，DataCollatorForSeq2Seq使用tokenizer分詞器來預處理輸入，但是它也會適應model。這是因為data_collator需要準備好解碼器的輸入ids，這個輸入ids是標籤的右移版本，在第一個位置上新增一個特殊的token。因為不同模型會有不同的移動方式，因此DataCollatorForSeq2Seq需要輸入model對象。

 from transformers import DataCollatorForSeq2Seq
 ​
 data_collator = DataCollatorForSeq2Seq(tokenizer, model=model)

我們可以測試一個例子看下效果。

指標

Seq2SeqTrainer在Trainer的基礎新增了使用generate()函數來進行評估和預測。在訓練期間，模型使用帶下三角注意力掩碼的decoder_input_ids來保證預測下一個token時，只能看到該token之前的token（平行計算，來加速訓練）。而在推理階段，不會使用這個操作，因為我們無法實現知道預測標籤，因此我們使用相同的設定來評估模型效果。

正如第1章所述，解碼器是一個token接一個token進行序貫預測的。在Transformers庫中，定義了generate()函數來完成這個操作。如果我們設定參數predict_with_generate=True，那麼我們Seq2SeqTrainer運行我們使用該函數完成模型評估。

翻譯任務的一個經典指標是BLEU得分，由Kishore Papineni et al. 在 a 2002 article這篇論文中提出。BLEU得分計算翻譯結果和標籤之間的相似關係。它無法衡量模型生成結果是否夠典雅或是否存在語法問題，而是使用統計規則來計算在生成文字中出現單詞是否也出現在標籤中。另外，對於在標籤中單詞重複出現進行一定的懲罰（防止模型輸出類似"the the the the the"），以及如果輸入句子長度比標籤短，也會進行懲罰（防止輸出句子只有一個單詞，例如the）。

BLEU指標的一個缺點是希望文字已經分詞了，這使得對於使用不同分詞器的模型之間計算得分變得困難。一般我們使用SacreBLEU作為翻譯結果的指標，克服了上述缺點。為了使用這個指標，首先需要安裝SacreBLEU庫，如下。

 !pip install sacrebleu

與第3章類似，我們使用函數evaluate.load()載入指標。

 import evaluate
 ​
 metric = evaluate.load("sacrebleu")

指標使用原始輸入和目標文字作為計算輸入，該函數可以接受多個標籤語句，這和常識想通（一個句子可以被翻譯成多個意思接近的句子）。我們使用的資料集只提供了一個翻譯結果，但是其他資料集有時候會提供多個標籤語句。因此預測值必須是一個語句序列，但是標籤必須是一個語句序列的序列（有兩個中括號）。

為了將模型輸出轉換為文字來計算指標，我們使用函數tokenizer.batch_decode()。同時，需要將標籤中的-100也給取代為補齊用的token。

 import numpy as np
 ​
 ​
 def compute_metrics(eval_preds):
     preds, labels = eval_preds
     # In case the model returns more than the prediction logits
     if isinstance(preds, tuple):
         preds = preds[0]
 ​
     decoded_preds = tokenizer.batch_decode(preds, skip_special_tokens=True)
 ​
     # Replace -100s in the labels as we can't decode them
     labels = np.where(labels != -100, labels, tokenizer.pad_token_id)
     decoded_labels = tokenizer.batch_decode(labels, skip_special_tokens=True)
 ​
     # Some simple post-processing
     decoded_preds = [pred.strip() for pred in decoded_preds]
     decoded_labels = [[label.strip()] for label in decoded_labels]
 ​
     result = metric.compute(predictions=decoded_preds, references=decoded_labels)
     return {"bleu": result["score"]}

微調模型

既然是官網教學，首先在訓練之前推薦你在官網登錄下，起始不註冊也是可以訓練，唯一區別就是在訓練過程中是否會把訓練過程記錄到Hugging Face Hub上。下面是登錄程式碼。

 from huggingface_hub import notebook_login
 ​
 notebook_login()

上面的程式碼返回一個登錄介面，可以輸入使用者名稱和密碼。

如果不是在notebook上運行程式碼，可以在命令列下運行下面的命令。

 huggingface-cli login

登錄之後，定義參數實例Seq2SeqTrainingArguments。該類對應序列到序列任務。在Seq2SeqTrainer的參數用上面的類來定義，如下。

 from transformers import Seq2SeqTrainingArguments
 ​
 args = Seq2SeqTrainingArguments(
     f"marian-finetuned-kde4-en-to-fr",
     evaluation_strategy="no",
     save_strategy="epoch",
     learning_rate=2e-5,
     per_device_train_batch_size=32,
     per_device_eval_batch_size=64,
     weight_decay=0.01,
     save_total_limit=3,
     num_train_epochs=3,
     predict_with_generate=True,
     fp16=True,
     push_to_hub=True,
 )

與TrainingArguments的常見參數（常見參數有學習率、訓練輪數、批大小和一些權重衰減）不同，有一些不一樣的參數，如下：

  • 沒有設定常規驗證機制，我們在整個模型訓練完成之後再進行驗證；
  • 設定fp16=True, 用來加速GPU的訓練；
  • 設定predict_with_generate=True，預測的時候使用generate()函數；
  • 設定push_to_hub=True，在每個epoch結束後將模型上傳到Hub上。

注意這裡可以通過設定參數hub_model_id來指定上傳到Hub的repo名稱（當然也可以在repo名前加上公司或組織名稱）。例如，我們上傳模型到 huggingface-course organization，可以在Seq2SeqTrainingArguments中設定參數hub_model_id="huggingface-course/marian-finetuned-kde4-en-to-fr"。默認情況下，一個repo使用你的namespace和輸出資料夾的名稱作為上傳repo的名稱，因此默認情況下你的上傳模型名稱為"sgugger/marian-finetuned-kde4-en-to-fr"。

如果輸出資料夾已經存在，那麼定義Seq2SeqTrainer時會報錯。糾正方法就是重新克隆一個repo到本地或者直接刪除。

最後，將參數傳入Seq2SeqTrainer。

 from transformers import Seq2SeqTrainer
 ​
 trainer = Seq2SeqTrainer(
     model,
     args,
     train_dataset=tokenized_datasets["train"],
     eval_dataset=tokenized_datasets["validation"],
     data_collator=data_collator,
     tokenizer=tokenizer,
     compute_metrics=compute_metrics,
 )

最後，呼叫函數push_to_hub()來將最終版本的模型上傳到Hub上。Trainer提供了一個默認的模型卡片（也就是上傳Hub後，看到的Readme），其中包含驗證資料集的驗證結果。同時也會在模型網頁右側，給出模型的推理demo，可以在這個demo上玩一下該模型。一般情況下，在模型類的介面上沒有必要做什麼操作，但是本例中，需要指定上傳模型的標籤為翻譯模型(因為類似的模型可以應用在所有類型的Seq2Seq問題上)。

 trainer.push_to_hub(tags="translation", commit_message="Training complete")

上述程式碼會返回commit的URL，方便我們定位模型網頁。

定製訓練過程（使用accelerate庫）

下面介紹使用accelerate庫來實現定製的訓練過程，適合喜歡定製訓練過程的小夥伴。

準備訓練要素

前面使用Trainer進行訓練已經準備好了訓練要素，這裡簡要介紹下。首先基於資料集建構DataLoader，然後將資料轉換為Pytorch格式（由data_collator實現）。

 from torch.utils.data import DataLoader
 ​
 tokenized_datasets.set_format("torch")
 train_dataloader = DataLoader(
     tokenized_datasets["train"],
     shuffle=True,
     collate_fn=data_collator,
     batch_size=8,
 )
 eval_dataloader = DataLoader(
     tokenized_datasets["validation"], collate_fn=data_collator, batch_size=8
 )

下面重初始化模型，來確保不是訓練Trainer之後的模型（為了觀察效果，微調最初下載的模型）。

 model = AutoModelForSeq2SeqLM.from_pretrained(model_checkpoint)

然後設定最佳化器（最近出的Lion聽說不錯，對很大的batch_size有用，對32左右的batch_size不知道效果是否會比adamw好）

 from transformers import AdamW
 ​
 optimizer = AdamW(model.parameters(), lr=2e-5)

準備好上面的DataLoader和最佳化器後，可以將這些要素送入函數accelerator.prepare()來做下wrapper。如果要在Colab上TPU進行訓練，則需要將所有的程式碼封裝為一個訓練函數，並且不要在notebook的cell中執行任何Accelerator實例化操作。

 from accelerate import Accelerator
 ​
 accelerator = Accelerator()
 model, optimizer, train_dataloader, eval_dataloader = accelerator.prepare(
     model, optimizer, train_dataloader, eval_dataloader
 )

現在我們將訓練資料載入器train_dataloader送入函數accelerator.prepare()，我們可以使用train_dataloader來計算整體的訓練步數（len(train_dataloader)是一個epoch的訓練步數）。需要注意的是在計算訓練步數，要在accelerate.prepare()之後進行。這是因為這個函數會改變dataloader的大小。另外，訓練使用經典線性學習率調度器（學習率按照線性比例從初始設定的學習率降低到0）。

 from transformers import get_scheduler
 ​
 num_train_epochs = 3
 num_update_steps_per_epoch = len(train_dataloader)
 num_training_steps = num_train_epochs * num_update_steps_per_epoch
 ​
 lr_scheduler = get_scheduler(
     "linear",
     optimizer=optimizer,
     num_warmup_steps=0,
     num_training_steps=num_training_steps,
 )

下步依然是huggingface的老操作了，登錄Hub，為了將訓練過程和模型參數上傳（如果wandb訪問過慢的話，這個其實也不錯）。建立repo，然後設定名稱。可以通過函數get_full_repo_name()來獲取repo的全名。

 from huggingface_hub import Repository, get_full_repo_name
 ​
 model_name = "marian-finetuned-kde4-en-to-fr-accelerate"
 repo_name = get_full_repo_name(model_name)
 repo_name
 'sgugger/marian-finetuned-kde4-en-to-fr-accelerate'

然後將repo克隆到本地。如果資料夾已經存在，那麼該資料夾必須是之前從hub上克隆下來的。

 output_dir = "marian-finetuned-kde4-en-to-fr-accelerate"
 repo = Repository(output_dir, clone_from=repo_name)

現在就可以通過函數repo.push_to_hub將output_dir設定的資料上傳。同時，會在每個訓練epoch結束時將模型參數上傳（當然，和模型訓練是兩個非同步處理程序，不會影響主訓練處理程序）。

訓練過程

下面編寫完整的訓練過程。為了簡化評估部分，定義postprocess()函數將預測結果和標籤轉換為字串（我們metric函數需要，也就是sacrebleu的輸入是兩個字串）

 def postprocess(predictions, labels):
     predictions = predictions.cpu().numpy()
     labels = labels.cpu().numpy()
 ​
     decoded_preds = tokenizer.batch_decode(predictions, skip_special_tokens=True)
 ​
     # Replace -100 in the labels as we can't decode them.
     labels = np.where(labels != -100, labels, tokenizer.pad_token_id)
     decoded_labels = tokenizer.batch_decode(labels, skip_special_tokens=True)
 ​
     # Some simple post-processing
     decoded_preds = [pred.strip() for pred in decoded_preds]
     decoded_labels = [[label.strip()] for label in decoded_labels]
     return decoded_preds, decoded_labels

訓練過程與之前章節類似（主要是model(**batch) -> loss -> loss.backward() -> optimizer.step()，即模型前饋和反饋過程，這是核心內容），下面主要介紹不同的地方。

第一個不同就是我們使用generate()函數來生成預測，這是我們基礎模型的一個函數，而不是Accelerate庫wrapper之後的模型的一個函數。因此在呼叫generate()函數之前，需要進行unwrapper操作。

第二個不同就是，類似token分類任務，需要同時對輸入和輸出進行補齊操作（這個好理解，就是一批的輸入和輸出資料都存在長度不相同的情況）。這裡在gather()函數之前呼叫accelerate.pad_across_processes()來保證輸入和輸出各自批次長度相同。如果不做上面的操作，評估過程會一致報錯（預測結果是否正確，我也可以使用精準率，就不需要了，但是精準率不靠譜）。

 from tqdm.auto import tqdm
 import torch
 ​
 progress_bar = tqdm(range(num_training_steps))
 ​
 for epoch in range(num_train_epochs):
     # 訓練過程
     model.train()
     for batch in train_dataloader:
         outputs = model(**batch)
         loss = outputs.loss
         accelerator.backward(loss)
 ​
         optimizer.step()
         lr_scheduler.step()
         optimizer.zero_grad()
         progress_bar.update(1)
 ​
     # 評估過程，使用generate解碼很慢，因此如果評估資料集很大，那麼就評估過程會非常慢
     # 具體的體驗可以參考uer-py中的一些模型訓練過程
     model.eval()
     for batch in tqdm(eval_dataloader):
         with torch.no_grad():
             generated_tokens = accelerator.unwrap_model(model).generate(
                 batch["input_ids"],
                 attention_mask=batch["attention_mask"],
                 max_length=128,
             )
         labels = batch["labels"]
 ​
         # 在gather之前進行pad，這一步不做程式碼執行階段會報錯
         generated_tokens = accelerator.pad_across_processes(
             generated_tokens, dim=1, pad_index=tokenizer.pad_token_id
         )
         labels = accelerator.pad_across_processes(labels, dim=1, pad_index=-100)
 ​
         predictions_gathered = accelerator.gather(generated_tokens)
         labels_gathered = accelerator.gather(labels)
 ​
         decoded_preds, decoded_labels = postprocess(predictions_gathered, labels_gathered)
         metric.add_batch(predictions=decoded_preds, references=decoded_labels)
 ​
     results = metric.compute()
     print(f"epoch {epoch}, BLEU score: {results['score']:.2f}")
 ​
     # Save and upload
     accelerator.wait_for_everyone()
     unwrapped_model = accelerator.unwrap_model(model)
     unwrapped_model.save_pretrained(output_dir, save_function=accelerator.save)
     if accelerator.is_main_process:
         tokenizer.save_pretrained(output_dir)
         repo.push_to_hub(
             commit_message=f"Training in progress epoch {epoch}", blocking=False
         )
 epoch 0, BLEU score: 53.47

 epoch 2, BLEU score: 54.44

使用微調模型

前面已經介紹了如何在Hub上使用微調模型進行推理操作。更優雅的方法是使用pipeline（這個工廠流水線的單詞還真是上頭！），需要指定模型的ID，具體如下。

 from transformers import pipeline
 ​
 # 可以替換model_checkpoint為你自己的translation模型ID
 model_checkpoint = "huggingface-course/marian-finetuned-kde4-en-to-fr"
 translator = pipeline("translation", model=model_checkpoint)
 translator("Default to expanded threads")
 [{'translation_text': 'Par défaut, développer les fils de discussion'}]

正如預期（好吧，結果論！），上面的例子翻譯結果還不來（我也不懂法語，誰知道賴不賴！），意思就是微調後，從預訓練的領域適應到了我們資料集所處的領域（所謂的完成了領域適應性訓練），也就是沒有不翻譯英文單詞threads，同樣plugin也是按照前面討論的進行了翻譯（在法文是一樣的意思和單詞拼寫！）

 translator(
     "Unable to import %1 using the OFX importer plugin. This file is not the correct format."
 )
